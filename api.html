API¶

Aerial Objects¶

This module provides some simple classes that are used to simulate flying drones and other objects.

Tip:
Altitudes are expected in meters, latitudes and longitudes are expected in degrees. By convention N and E are positive angles and S and W are negative angles.

See also

https://en.wikipedia.org/wiki/World_Geodetic_System

class dronedirector.aerial.AerialObject(altitude, latitude, longitude)[source]¶

Base class for aerial objects.

Aerial objects can be static or dynamic but must define something about their location (latitude, longitude) and how tall they are/how high up they fly. An example of a static aerial object (like a tree or building) is given below. Note that at a minimum,altitude, latitude, and longitude should be defined.

class CaliRedwood(AerialObject):
    # Example of a completely static object
    def __init__(self):
        super(CaliRedwood, self).__init__(altitude=itertools.cycle([100.0]),
                                          latitude=itertools.cycle([37.8716]),
                                          longitude=itertools.cycle([-122.2727]))

Distance Calculation Utilities¶

This module provides functions for determining distances between aerial objects (static and dynamic) which are near the surface of planet Earth. This module makes use of compilation to speed up mathematical operations.

Tip:
Altitudes are expected in meters, latitudes and longitudes are expected in degrees. By convention N and E are positive angles and S and W are negative angles.

See also

https://en.wikipedia.org/wiki/World_Geodetic_System

dronedirector.distance.earth_radius(lat)[source]¶

Compute the radius of the Earth for a given latitude.

The Earth isn’t exactly spherical; depending on what latitude an object is at, the distance from sea-level to the center of the Earth varies slightly.

Parameters:lat (float) – Latitude in radians
Returns:radius (float) – Approximate distance to sea level at the current latitude in meters

Kafka Producer¶

This module provides a simple interface for generating messages from Drone messages. If a schema registry is not available, JSON messages can be sent using fly_drones().

dronedirector.producer.fly_avro_drones(bootstrap_servers, schema_registry_url, nmessages, default_value_schema_str='\n{\n "namespace": "dronedirector.drones", \n "name": "drone", \n "type": "record", \n "fields": [\n {\n "name": "uid", \n "type": "string"\n }, \n {\n "name": "region", \n "type": "string"\n }, \n {\n "name": "dronetime", \n "type": "string"\n }, \n {\n "name": "altitude", \n "type": "double"\n }, \n {\n "name": "latitude", \n "type": "double"\n }, \n {\n "name": "longitude", \n "type": "double"\n }\n ]\n}', producer_dict_kwargs=None, topic_name='drones_raw', time_delay=0, drones=None)[source]¶

A simple example of sending structured messages from drones to a message broker.

Parameters:
  • bootstrap_servers (str) – Comma separated string of Kafka servers
  • schema_registry_url (str) – Schema registry URL
  • nmessages (int) – Number of messages to send
  • default_value_schema_str (str) – String Avro schema compatible with mdrone messages
  • producer_dict_kwargs (dict) – Optional keyword arguments for producer
  • topic_name (str) – Topic name to which drone messages will be sent
  • time_delay (int) – Delay time between cycles when producing messages
  • drones (iterable) – Iterable of drones from which to generate messages
Tip:
Schemas should match the messages sent by drones.
